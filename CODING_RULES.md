# Coding Rules

이 문서는 이 프로젝트에서 코드를 작성하거나 수정할 때 항상 적용하는 기준이다. 새 기능을 만들 때도, 기존 코드를 손볼 때도 아래 규칙을 우선한다.

## 1. Tailwind 규칙

- Tailwind 유틸리티를 기본으로 사용한다.
- `w-full`, `p-4`, `gap-2`, `rounded-md`, `text-sm`처럼 Tailwind 스케일에 있는 값을 먼저 고른다.
- `w-[123px]`, `p-[13px]`, `text-[#307cff]`, `bg-[rgba(...)]`, `style={{ ... }}` 같은 arbitrary value와 인라인 스타일은 최대한 피한다.
- 예외는 고정 디바이스 프레임, 외부 디자인 시안의 정확한 좌표, SVG/캔버스/아이콘 내부처럼 Tailwind 스케일로 표현하기 어려운 경우에만 허용한다.
- 예외 값이 반복되면 그대로 복사하지 말고 디자인 토큰이나 공용 컴포넌트로 올린다.
- 새 색상은 바로 hex로 쓰지 않고 `app/globals.css`의 CSS 변수와 `@theme inline` 토큰에 먼저 추가할 수 있는지 검토한다.

## 2. 디자인 시스템 우선

- 현재 디자인 시스템의 시작점은 `app/globals.css`의 토큰과 `components/`의 공용 컴포넌트다.
- 새 화면을 만들기 전에 기존 컴포넌트를 먼저 확인한다: `AppFrame`, `MobileHeader`, `BottomNav`, `ScreenSection`, `UiCard`, `SettingsListItem`, `ActionButton`, `ProfileAvatar`, `Toggle`, `AssetIcon`, `figma-controls`.
- 메뉴, 설정, 내 활동처럼 아이콘과 제목/설명으로 구성된 내비게이션 리스트는 `SettingsListItem`을 우선 사용한다.
- 기존 컴포넌트 조합으로 해결할 수 있으면 새 컴포넌트를 만들지 않는다.
- 새 컴포넌트는 실제로 반복되거나, 기존 컴포넌트의 책임을 흐리게 만들 정도로 새로운 UI 패턴이 필요할 때만 추가한다.
- 새 컴포넌트를 추가할 때는 토큰, spacing, radius, typography, 상태 스타일을 기존 시스템과 맞춘다.
- 공용 컴포넌트는 `components/`에 두고, 특정 라우트에서만 쓰는 UI는 해당 라우트 가까이에 둔다.

## 3. Next.js App Router 기준

- 이 프로젝트는 Next.js `16.2.9`와 App Router를 사용한다. 코드 작성 전 관련 문서는 `node_modules/next/dist/docs/`에서 확인한다.
- `page.tsx`, `layout.tsx`, `route.ts` 등 App Router 파일 규칙을 따른다.
- 기본은 Server Component로 작성하고, 상태, 이벤트 핸들러, 브라우저 API, custom hook이 필요할 때만 `"use client"`를 붙인다.
- `"use client"`는 가능한 작은 컴포넌트 경계에만 둔다.
- API 라우트는 `app/api/**/route.ts` 패턴을 유지하고, 응답 구조와 에러 처리를 일관되게 작성한다.

## 4. TypeScript와 컴포넌트 작성

- props 타입은 명시한다. 여러 곳에서 공유될 때만 export한다.
- 컴포넌트는 작고 읽기 쉽게 유지한다. UI, 데이터 처리, 상태 로직이 커지면 역할별로 분리한다.
- 문자열 union으로 표현할 수 있는 variant는 자유 문자열보다 union 타입을 사용한다.
- `any`는 피하고, 외부 데이터는 필요한 모양으로 좁혀서 사용한다.
- 불필요한 주석은 쓰지 않는다. 복잡한 의도나 예외가 있을 때만 짧게 남긴다.

## 5. UI와 접근성

- 버튼, 링크, 입력 등 인터랙션 요소는 의미에 맞는 HTML 요소를 사용한다.
- 클릭 가능한 아이콘에는 접근 가능한 이름을 제공한다.
- 계정은 팬/작가 타입으로 분리해 표시하지 않는다. 창작자 기능은 누구나 시작할 수 있는 도구로 노출한다.
- 구독 멤버십 진입점은 만들기와 관리를 동시에 노출하지 않고, 생성 상태에 따라 단일 항목의 라벨을 바꾼다.
- 모바일 기준 레이아웃에서도 텍스트가 잘리지 않고 겹치지 않도록 확인한다.
- 로딩, 비어 있음, 오류 상태는 사용자가 다음 행동을 알 수 있게 만든다.
- 색상만으로 상태를 전달하지 않는다.

## 6. 변경 범위와 검증

- 요청과 직접 관련 없는 리팩터링은 하지 않는다.
- 기존 동작을 바꾸는 경우, 영향을 받는 화면과 API를 함께 확인한다.
- 의미 있는 코드 변경 후에는 가능한 한 `npm run lint`를 실행한다.
- 라우팅, 빌드 설정, 서버/클라이언트 경계, API 변경이 있으면 `npm run build`까지 확인한다.
- 기존 파일을 수정할 때는 주변 스타일과 패턴을 먼저 읽고 맞춘다.

## 7. 새 의존성 기준

- 새 패키지는 꼭 필요할 때만 추가한다.
- 브라우저 번들 크기, 유지보수성, 기존 코드로 대체 가능한지를 먼저 검토한다.
- UI 라이브러리나 스타일 도구를 추가하기보다 현재 Tailwind와 컴포넌트 시스템을 우선한다.
